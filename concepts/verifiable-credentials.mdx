title: "Verifiable credentials"

Verifiable credentials let an issuer make claims about a holder that a verifier can check later. Didit ID implements this model with standards-based issuance, wallet presentation, selective disclosure, revocation checks, and issuer trust checks.

Didit ID uses OpenID4VCI 1.0 for issuance, OpenID4VP 1.0 with DCQL for verification, and SD-JWT VC draft-16 for the credential format. Credentials use the `application/dc+sd-jwt` media type.

## The issuer, holder, and verifier model

<CardGroup cols={3}>
  <Card title="Issuer" icon="signature" href="/api-reference/openid4vci/issuers-slug-openid-credential-issuer-get">
    The issuer creates a credential, signs it with an issuer key, and publishes metadata and JWKS under `/issuers/{slug}/.well-known/*`.
  </Card>
  <Card title="Holder" icon="wallet" href="/api-reference/openid4vci/credential-post">
    The holder receives the credential, keeps the disclosures, controls the holder key, and chooses which claims to present.
  </Card>
  <Card title="Verifier" icon="shield-check" href="/api-reference/verification/presentations-uuid-response-post">
    The verifier requests specific claims, receives a presentation, and checks cryptographic proof, status, and trust.
  </Card>
</CardGroup>

The triangle works because each role has a separate responsibility. The issuer signs the original credential. The holder proves possession of the bound holder key. The verifier checks the issuer signature, the holder binding, the intended audience, the nonce, expiry, revocation status, and issuer trust.

## Where Didit ID fits

Didit ID gives you the credential infrastructure around the open protocols:

- **Credential design**: Create schemas and templates with `vct` values and selectively disclosable attributes through [`/credential-schemas`](/api-reference/credentials/credential-schemas-post) and [`/credential-templates`](/api-reference/credentials/credential-templates-post).
- **Issuance**: Create credential offers with [`POST /credential-offers`](/api-reference/openid4vci/credential-offers-post), then let a wallet redeem the pre-authorized code through the OpenID4VCI token, nonce, and credential endpoints.
- **Verification**: Create DCQL-backed presentation requests with [`POST /presentations/request`](/api-reference/verification/presentations-request-post), then verify `vp_token` responses with [`POST /presentations/{uuid}/response`](/api-reference/verification/presentations-uuid-response-post).
- **Lifecycle**: List issued credentials and revoke, suspend, or reactivate them through the credential management APIs.
- **Trust**: Register trusted issuers and relying parties, then enforce trusted-issuer checks during verification.
- **Issuer keys**: Rotate issuer signing keys with overlapping validity so previously issued credentials can remain verifiable until the rotated key expires.
- **Status lists**: Publish whole Token Status Lists so verifiers can check revocation without fetching one credential-specific status record.

<Note>
  Management endpoints use `Authorization: Bearer $DIDIT_API_KEY`. Holder and verifier protocol endpoints such as `/oauth/token`, `/nonce`, `/credential`, `/presentations/{uuid}`, and `/presentations/{uuid}/response` are public.
</Note>

## Issuance flow

Issuance starts with the issuer and ends with a holder-controlled credential.

<Steps>
  <Step title="Define the credential type">
    Create a schema with a `vct` such as `AirlineLoyalty`. Mark fields as selectively disclosable when the holder should be able to reveal them independently.
  </Step>
  <Step title="Create a credential offer">
    Call [`POST /credential-offers`](/api-reference/openid4vci/credential-offers-post) with a `template_id` and the claims to issue. Didit ID returns an `offer_id`, a `credential_offer_uri`, a `pre_authorized_code`, and a `tx_code`.
  </Step>
  <Step title="Let the wallet redeem the offer">
    The holder wallet exchanges the `pre_authorized_code` and `tx_code` at [`POST /oauth/token`](/api-reference/openid4vci/oauth-token-post), requests a `c_nonce` at [`POST /nonce`](/api-reference/openid4vci/nonce-post), and sends a holder proof JWT to [`POST /credential`](/api-reference/openid4vci/credential-post).
  </Step>
  <Step title="Issue the SD-JWT VC">
    Didit ID returns a credential string such as `<jws>~<disclosure>~` and the issued `vct`. The signed JWS contains digest references for selectively disclosable claims. The holder stores the disclosures needed to reveal those claims later.
  </Step>
</Steps>

You can also issue directly through [`POST /credentials/issue`](/api-reference/credentials/credentials-issue-post) when your backend already has the holder public key as `holder_jwk`.

<CodeGroup>

```bash cURL
curl -X POST "http://localhost:8011/v1/credential-offers" \
  -H "Authorization: Bearer $DIDIT_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "template_id": "<uuid>",
    "claims": {
      "given_name": "Ada",
      "tier": "gold"
    }
  }'
```

```javascript JavaScript
const response = await fetch("http://localhost:8011/v1/credential-offers", {
  method: "POST",
  headers: {
    Authorization: `Bearer ${process.env.DIDIT_API_KEY}`,
    "Content-Type": "application/json",
  },
  body: JSON.stringify({
    template_id: "<uuid>",
    claims: {
      given_name: "Ada",
      tier: "gold",
    },
  }),
});

const offer = await response.json();
```

</CodeGroup>

## Selective disclosure

Selective disclosure lets the holder reveal only the requested claims. The issuer signs digest commitments to hidden claims. The holder keeps the disclosure values. During presentation, the holder sends the issuer-signed credential plus only the disclosures requested by the verifier.

For example, a loyalty credential can contain `given_name`, `family_name`, `date_of_birth`, and `tier`. A verifier can request only `tier`. The presentation response then returns `disclosed_claims` with `tier`, while the other claims remain undisclosed.

<ParamField body="requested_claims" type="string[]">
  The claim names the verifier asks for when creating a presentation request.
</ParamField>

<ResponseField name="disclosed_claims" type="object">
  The verified claims revealed by the holder in the submitted `vp_token`.
</ResponseField>

<ResponseField name="checks" type="object">
  The verification result for signature, holder key binding, audience, nonce, algorithm allowlist, expiry, and revocation.
</ResponseField>

## Verification flow

Verification starts with a relying party and ends with a verdict.

<Steps>
  <Step title="Create a presentation request">
    Call [`POST /presentations/request`](/api-reference/verification/presentations-request-post) with `requested_vct`, `requested_claims`, `aud`, and optional `trusted_iss`. Didit ID builds a DCQL query and nonce.
  </Step>
  <Step title="Send the request to the wallet">
    Deliver the OpenID4VP request to the wallet. Browser-based flows can use the W3C Digital Credentials API with `openid4vp-v1-signed`.
  </Step>
  <Step title="Submit the presentation">
    The wallet returns a `vp_token` to [`POST /presentations/{uuid}/response`](/api-reference/verification/presentations-uuid-response-post). The presentation includes the selected disclosures and a key-binding JWT.
  </Step>
  <Step title="Read the verdict">
    Didit ID returns `verdict`, `disclosed_claims`, and `checks`. You can also poll [`GET /presentations/{uuid}`](/api-reference/verification/presentations-uuid-get) or read the verification log through [`GET /verifications`](/api-reference/verification/verifications-get).
  </Step>
</Steps>

<CodeGroup>

```bash cURL
curl -X POST "http://localhost:8011/v1/presentations/request" \
  -H "Authorization: Bearer $DIDIT_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "requested_vct": "AirlineLoyalty",
    "requested_claims": ["tier"],
    "aud": "checkin-rp",
    "trusted_iss": ""
  }'
```

```javascript JavaScript
const response = await fetch("http://localhost:8011/v1/presentations/request", {
  method: "POST",
  headers: {
    Authorization: `Bearer ${process.env.DIDIT_API_KEY}`,
    "Content-Type": "application/json",
  },
  body: JSON.stringify({
    requested_vct: "AirlineLoyalty",
    requested_claims: ["tier"],
    aud: "checkin-rp",
    trusted_iss: "",
  }),
});

const presentationRequest = await response.json();
```

</CodeGroup>

## Why SD-JWT VC

SD-JWT VC gives Didit ID a compact, web-native credential format with selective disclosure built in. It works well with OpenID4VCI and OpenID4VP because the credential is a signed JWT, while hidden claims are revealed through separate disclosure values only when the holder chooses to present them.

Didit ID uses SD-JWT VC because it supports:

- **Selective disclosure** through `_sd` digest commitments and holder-provided disclosures.
- **Holder key binding** through the `cnf` claim and presentation-time KB-JWT checks.
- **Credential typing** through the `vct` claim.
- **Revocation and suspension checks** through IETF Token Status List status entries.
- **Issuer key rotation** through `kid`-pinned issuer signatures and overlapping validity for rotated keys.

<Warning>
  mDL, ISO 18013-5/-7, and Apple or Google Wallet support are documented as fast-follow capabilities. Use the SD-JWT VC, OpenID4VCI, and OpenID4VP endpoints described here for live integrations today.
</Warning>

## Verification checks

When a wallet submits a presentation, Didit ID evaluates the checks returned by [`POST /presentations/{uuid}/response`](/api-reference/verification/presentations-uuid-response-post):

- `signature`: The issuer signature is valid for the issuer key.
- `key_binding`: The holder proves possession of the key bound in the credential.
- `aud`: The presentation is intended for the requested audience.
- `nonce`: The presentation is bound to the nonce from the request.
- `alg_allowlist`: The credential and proof use allowed algorithms.
- `not_expired`: The credential is still within its validity window.
- `not_revoked`: The credential is active according to its Token Status List entry.

If the request includes `trusted_iss`, Didit ID also evaluates the issuer against the tenant trust registry.
